# Бърз старт

Целева версия: **v1.2.9** (2026-03-15)

Това ръководство обхваща двата поддържани работни потока:
- **Thonny поток** (директен firmware терминал)
- **Lab поток** (`runner/lab.py`, рецепти + графики + метрики + папки с резултати)

Използвайте документа като оперативна отправна точка за студенти и преподаватели.

## 1. Предварителни условия

### Хардуер

- RP2040 платка (клас Pico)
- PicoPID термичен обект, свързан към платката
- USB кабел

### Софтуер на компютъра

- Python 3.10+
- Thonny (препоръчителен за интерактивна работа с firmware): https://thonny.org
- Python пакети:

```bash
pip install pyserial pyyaml matplotlib numpy
```

### Firmware файлове

Качете всички файлове от:
- `firmware/`

в root файловата система на платката.

---

## 2. Еднократна настройка на платката в Thonny

1. В Thonny изберете интерпретатор: `MicroPython (Raspberry Pi Pico)`.
2. Изберете правилния сериен порт.
3. Качете всички firmware файлове от `firmware/`.
4. Стартирайте `firmware/main.py`.
5. Изчакайте firmware подканата `# READY:`.

След това платката може да се използва както от Thonny, така и от `runner/lab.py`.

---

## 3. Поток A: Thonny (директен firmware терминал)

Използвайте този поток при директна интерактивна работа с firmware без автоматизация по рецепти.

Типична последователност:

1. `check`
2. `status`
3. `params` / `params pid` / `params model` / `params tune`
4. команда за изпълнение:
   - `control` (затворен контур)
   - `tune` (настройка по избрано `TUNING_METHOD`)
   - `model` (FOPDT идентификация)
   - `monitor` (пасивно наблюдение, нагревател OFF)
5. при активен run: `stop`

Синтаксис за промяна на параметър:
- `<PARAM> <VALUE>` (основен)
- `<PARAM>=<VALUE>` (също се приема)

Примери:

```text
SETPOINT_C 45
TUNING_METHOD CC_PID
PID_ALGORITHM PARALLEL
```

Собственост на изхода:
- Firmware статус/репорт токените винаги започват с `# ...`
- Телеметрията е без префикс:
  - `PV:... SP:... OP:...`
  - В MPC режим се добавят `YH` и `YP`

---

## 4. Поток B: Lab Runner (`runner/lab.py`)

Използвайте този поток за повторяеми упражнения, папки с резултати, CSV, live графика и live метрики.

Стартиране от root на хранилището:

```bash
python3 runner/lab.py
```

(`python3 lab.py` също работи, ако текущата директория е `runner/`.)

След connect/sync host влиза директно в `lab>`.

### Host команди (`lab>`)

- `h` help
- `c` catalog
- `e <id|exp_id>` стартиране на експеримент
- `s` спиране на активен run
- `k` firmware `check`
- `u` host статус
- `x` reconnect на serial сесията
- `b` terminal home/refresh
- `q` изход

Вход, който не е разпознат като host команда, се препраща към firmware.

### Управление при активен run

При активни дълги изпълнения (`control`, `tune`, `model`, `monitor`):
- `s` спира текущото изпълнение
- `h` показва наличните run команди

### Собственост на съобщенията

- Host редовете са с префикс `LAB:`
- Firmware редовете запазват `# ...` и телеметричния формат

### Артефакти от изпълнение

Всеки run се записва в:
- `runner/runs/<timestamp>__<EXPERIMENT>__<shortname>/`

Типично съдържание:
- CSV телеметрия
- log текст
- metrics JSON
- графики

---

## 5. Система с рецепти (`runner/lab.yaml`)

`runner/lab.yaml` е host-side source of truth за упражненията.
`firmware/config.py` е fallback за параметри, които не са override-нати от рецепта.

Видове рецепти:
- `standard` run с фиксирана продължителност
- `fopdt` open-loop идентификация на модел
- `tuning` tuning процедура
- `sweep` cartesian параметричен sweep

Токен за завършване, използван от host синхронизацията:
- `# FINISH: done`

---

## 6. Препоръчителен студентски поток

1. `model` за идентификация/обновяване на FOPDT модел (`MODEL_*` в RAM)
2. `tune` по избрано `TUNING_METHOD` (model-based или relay-based)
3. `control` с обновените коефициенти
4. Сравнение на PV/SP/OP трендовете и запазените метрики

Повтаряйте с промяна само на един параметър.

---

## 7. Безопасност и валидация

Преди всяко изпълнение:

1. `check`
2. Потвърдете:
   - `# RESULT: check passed (config is valid)`
3. Проверете safety настройките в `config.py`:
   - `TEMP_CUTOFF_C`

При cutoff или фатално условие:
- нагревателят се изключва принудително
- firmware извежда warning/error токени

---

## 8. Отстраняване на проблеми

### Няма serial връзка

- Преподключете USB
- Проверете интерпретатора/порта в Thonny
- Използвайте `x` в `lab>` за reconnect

### Host се свързва, но run не стартира

- Изпълнете `k` (firmware check) от `lab>`
- Изпълнете `params` и проверете настройките за избрания режим

### Прозорецът за графика/метрики не остава отворен

- Уверете се, че има активен run (`control`, `model`, `tune`, `monitor` или `e <id>`)
- Не затваряйте прозорците преди стартиране на run

### Неочаквано поведение на команди

- Използвайте `h` в `lab>`
- Използвайте firmware командата `help` за device-side списък с команди
